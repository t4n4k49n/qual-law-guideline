以下は、**対象リスト（国内法令〜EU/FDA/PIC/S/WHOまで）すべてに同一フォーマットで適用できる「中間構造（IR: Intermediate Representation）」の提案です。
目的（DQのGMPチェックシートに“選択肢として”提供し、選択した条文/記載単位を文脈付きで提示できる）に合わせて、「条文ツリー（細切れ可能）」＋「文脈呼び出し」＋「選択/グルーピング設定（親の選び方）」**を YAML で表現できる構造にしています。

1) 提案の全体像（3レイヤー構成）

A. RegDoc（規制文書）本体：共通IR

  • 法令・省令・規則・ガイドライン・マニュアル等を “ノードの木構造” に正規化

  • 章/節/条/項/号/イロハ/paragraph(a)…などを全部「node.kind」で吸収

  • 最小単位（チェックシートに載せたい粒度）は、必要なら statement（文章/要件単位） まで下げて分割できる

B. プロファイル（用途別設定）：DQのGMPチェックシート用

  • 「どの kind を選択可能にするか（細切れ粒度）」

  • 「選択したノードを、どの親でグルーピング（見出し化）するか」

  • 「選択ノードの表示時に、どこまで先祖を表示するか（文脈）」
    を ドキュメント単位で設定できる

C. メタ情報（施行日・所管など）

  • 文書レベル（doc）に保持（ご要望どおり“正規化と直結しないが YAML に含める”）

2) 共通IR（RegDoc YAML）の提案スキーマ

“スキーマ”は YAML で擬似的に示します（実装ではこの構造を JSON Schema 等で検証すると堅いです）。
ここでは 全対象（e-Gov法令、PDFガイドライン、EU GMP、CFR、PIC/S、WHO）を同型で表現できることを重視しています。

yaml
schema: qai.regdoc_ir.v1

doc:
  id: <string>                  # 一意な文書ID（例: jp_mhlw_gmp_ordinance）
  title: <string>               # 正式名称
  short_title: <string|null>    # 略称（例: GMP省令 / 21 CFR Part 211）
  jurisdiction: <string>        # JP / EU / US / INT など
  doc_type: <string>            # statute / cabinet_order / ministerial_ordinance / rule /
                                # guideline / manual / standard / annex / other
  language: <string>            # ja / en など
  issuer:
    name: <string|null>         # 厚労省 / PMDA / EC / FDA / PIC/S / WHO 等
    kind: <string|null>         # ministry / agency / international_org / other
  identifiers:                  # 文書を特定する外部ID（あるものだけ入れる）
    e_gov_law_id: <string|null>
    law_number: <string|null>
    cfr_title: <int|null>
    cfr_part: <int|null>
    eu_volume: <string|null>
    pics_doc_id: <string|null>
    who_publication_id: <string|null>

  version:
    as_of: <date>               # 収録版の日付（e-Govの改正日や、PDF版の日付）
    effective_from: <date|null> # 施行日（分かる場合）
    effective_to: <date|null>
    revision_note: <string|null>

  sources:                      # 取得元（複数可：EU GMPは章PDFが分割など）
    - url: <string>
      format: <string>          # html / pdf / doc / other
      retrieved_at: <date>      # 取得日
      checksum: <string|null>   # 任意（sha256等）
      label: <string|null>      # "chapter 1" 等

  legal_meta:                   # 所管や根拠法、位置づけ（任意）
    competent_authority: [<string>]
    legal_basis: [<string>]
    binding_level: <string|null>  # binding / guidance / mixed / unknown
    license_note: <string|null>   # 再配布条件や注意（WHO等で重要）

content:
  nid: root
  kind: document
  num: null
  heading: null
  text: null
  role: structural               # structural / normative / informative / definition / note
  children: [ ... nodes ... ]

# 任意：ノード検索や参照の補助（生成しても良い）
index:
  # nid -> 最短表示名など（任意）
  display_name_by_nid: {}

3) ノード（node）設計：条文・節・段落・箇条書きを“全部同じ箱”で扱う

各 node は、次のフィールドを持つ想定です（必要なものだけ使う）。

yaml
nid: <string>            # 文書内で一意なノードID（パス形式推奨）
kind: <string>           # 正規化した種別: part/chapter/section/article/paragraph/item/subitem/point/annex/appendix/statement/table...
kind_raw: <string|null>  # 原文の種別表記: "条" "項" "号" "§" "Chapter" 等
num: <string|null>       # 表示用番号: "第十二条" "二" "ロ" "§211.22" "(a)" "3.2" 等
ord: <int|null>          # 並び順（ソート用）
heading: <string|null>   # 見出し（括弧書き見出し、章題など）
text: <string|null>      # 本文（chapeau/要件文/段落本文など）
role: <string>           # structural/normative/informative/definition/note
normativity: <string|null> # must/should/may/info（ガイドラインに有用）
tags: [<string>]         # 後で設備/無菌/CSV等と紐づけるためのタグ
refs:                    # 相互参照（任意）
  internal: [<nid>]
  external:
    - doc_id: <string>
      nid: <string>
source_spans:            # 出典位置（PDFページやHTMLアンカー等、任意だが強い）
  - source_label: <string|null>
    locator: <string|null>  # "p.12" "page=5" "article=12" 等、自由
children: [ ... ]

4) DQのGMPチェックシート用プロファイル（親の選び方・文脈の呼び出し）

ご要望の「文書ごと、かつ選択レベル（kind）ごとに“どの親を選ぶ/表示するか”」は、profilesとして別 YAML に切り出すのが運用しやすいです（文書が改訂されてもプロファイルは比較的保ちやすい）。

プロファイル例（1文書に対する設定）

yaml
schema: qai.regdoc_profile.v1

doc_id: jp_mhlw_pharmacy_structure_rules   # doc.id を参照

profiles:
  dq_gmp_checklist:
    selectable_kinds:
      # チェックシートの「選択肢」としてユーザーが選べる最小単位
      - subitem
      - item
      - statement

    grouping_policy:
      # 選択されたノードを、チェックシート上でどの親の下にぶら下げるか
      - when_kind: subitem
        group_under: item
      - when_kind: item
        group_under: article
      - when_kind: statement
        group_under: paragraph

    context_display_policy:
      # 「単位表記だけでは意味が分からない」問題への対策
      # 選択ノードを表示する際に、どこまで先祖（親/祖先）を同時表示するか
      - when_kind: subitem
        include_ancestors_until_kind: article   # articleまで遡って表示（見出し＋chapeau込み）
        include_headings: true
        include_chapeau_text: true
      - when_kind: statement
        include_ancestors_until_kind: section
        include_headings: true
        include_chapeau_text: false

    render_templates:
      # 出力（チェックシート）の表示テンプレ（任意）
      checklist_item_label: "{cite}"
      checklist_item_text: "{text}"
      context_block_format: "breadcrumb+snippets"

  • selectable_kinds：細切れ粒度のコントロール（文書ごとに違ってOK）

  • grouping_policy：チェックシートの階層（どの親を見出しにするか）

  • context_display_policy：表示時に呼び出す先祖範囲（「ロ」単体問題を解決）

7) このIRで満たせる要件（ご提示3点への対応）

✅ 1) チェックシートに載せる単位へ細切れ

  • kind を増やせる（article/paragraph/item/subitem/statement…）

  • 章節条の番号が無いガイドラインでも、段落や箇条書きを kind: point、必要なら文単位を kind: statement にして落とせる

✅ 2) 先祖（親/祖先）を呼び出せる

  • children による木構造 + nid で参照可能

  • 実装では「選択ノードから root まで parent を辿る」だけで祖先を復元できる

  • どの祖先まで表示するかは プロファイルで制御

✅ 3) 親の選び方（どの親でグルーピングするか）を設定できる

  • grouping_policy により「subitem は item の下」「item は article の下」などを文書別に定義可能

  • さらに「選択 kind ごとに文脈表示の深さ」を変えられる（例：statementはsectionまで、subitemはarticleまで）

8) 実務上の“推奨”オプション（将来効いてくる）

必須ではないですが、DQ/GMPチェックシート運用を安定させるなら入れておくと後で助かる項目です。

  • source_spans（出典位置）
    PDFベース（PMDA/WHO/EU等）で「この条文は何ページか」を残すと監査/根拠提示が楽

  • binding_level / normativity
    “法令（must）”と“ガイド（should）”を混在させるときに UI/出力で差が出せる

  • license_note
    WHOや一部ガイドラインは再配布条件が絡む可能性があるため、メタとして明示しておく
