「フォーマット別（txt/pdf）」だけで分けると、結局“文書系列の癖”がコードに散っていきがちです。そこで、**コードは共通基盤＋ファミリ別プロファイル（YAML）**の二層を基本にして、PDFだけは“前処理層”として別扱いにします。
（4ファイル運用＝meta / parser_profile / regdoc_ir / regdoc_profile の役割分担、という既存の設計思想にも沿います）

Code Set A：共通基盤（txt/pdf共通の“行→構造”エンジン）

  • 入力：テキスト行列（＝PDFも最終的に行へ落とす）

  • 出力：regdoc_ir（木構造）＋4ファイル一括生成の枠組み

  • 中核：

# 変更タスク詳細

## 1. profile_loader 拡張
- `load_parser_profile()` に以下を追加

  - `path: Optional[Path] = None`（既存互換）

- 例：`pics_part1_default_v1`, `pics_annex1_default_v1`

## 2. cli.py 拡張
- `bundle` コマンドに以下の option を追加
  - `--parser-profile PATH`（任意）
  - `--parser-profile-id TEXT`（任意）
  - `--regdoc-profile PATH`（任意）または `--context-root-kind TEXT`（任意）
  - `--jurisdiction` / `--language`（任意、未指定ならプロファイルから推定）
- 互換性：未指定なら現状どおり CFR default で動くこと
- meta.yaml の doc.jurisdiction, doc.language, sources.label などは CFR 固定にせず、
  option かプロファイル値（例: applies_to.jurisdiction, title 等）から埋められるようにする。
- meta の sources.url / retrieved_at は None のままでも良いが、
  少なくともスキーマ的に必須なら CLI で required にする（テストで検出できるようにする）。

## 3. text_parser.py 最小拡張
### 3-1) num 抽出のfallback
- いまは `m.groupdict().get("num")` しか拾っていない。
- 次の順で num を決める関数を追加して MarkerMatch に入れる：
  1) groupdict["num"] があればそれ
  2) それ以外の groupdict の最初の non-empty 値（例: alpha, n, roman）
  3) それでも無ければ None
- さらに marker 定義に `num_group` があればそれを最優先にする（任意機能）。

### 3-2) structural_kinds をプロファイル化
- parser_profile のトップに `structural_kinds: [...]` を許容（無ければ既存デフォルト）
  - CFR: ["part","subpart","section"]
  - PIC/S: ["chapter"] など（必要に応じて）
- NodeFactory の role/normativity、見出し(heading)扱いを structural_kinds に従わせる
  - structural => heading 優先 / role="structural" / normativity=None
  - non-structural => text 連結 / role="normative" / normativity="must"（現状維持でOK）
- 既存CFRテストが通ることを最優先。

## 4. PIC/S parser_profile YAML を追加
- `src/qai_text2ir/profiles/` に以下を追加

  - `pics_annex1_default_v1.yaml`
- marker_types と structure は「kind名ベース」で設計する（text_parser は kind で親判定しているため）
- 例（あくまで叩き台、fixture を見て調整）：
  - CHAPTER 5 -> kind: chapter, num: 5
  - 5.1. -> kind: paragraph (または section) / num: 5.1 など

- compound_prefix は CFR ほど重要でなければ max_depth=1 でよい。
- 重要：PIC/S の “番号付き本文” を heading に入れてしまうと本文が消えるので、
  structural_kinds と kind 設計で「本文は text に入る」ようにする。

# 詳細要件

## A. CLIで parser_profile を指定できるようにする
- `text2ir bundle` にオプションを追加:
  - `--parser-profile <path>` : 任意のYAMLを読み込む
  - 省略時は従来どおりデフォルト（us_cfr_default_v1.yaml）
- `profile_loader.load_parser_profile()` は path 指定が来たらそれを優先する（既に引数はあるのでCLIから渡す）

## B. num_group 対応
- 現状 text_parser は正規表現の group 名が `num` 固定になっている
- marker_types の各要素に `num_group` があれば、その group 名を使って num を抽出する
  - 例: num_group: "roman" なら m.group("roman")
- 互換性維持:
  - num_group が無い場合は従来通り "num" を使う
  - group が存在しない/取れない場合は num=None で落ちないこと

## C. CFR注記行（最低限）
- CFRにはセクション末尾に以下が頻出する:
  - `[43 FR 45077, Sept. 29, 1978, ...]` のようなFR引用
  - `Authority:` `Source:` のようなメタ行
- これらが「直前の(4)などの末端ノード」に連結されると意味が崩れるので、構造として分離する
- 実装方針（最小でOK）:
  - us_cfr_default_v1.yaml に marker_types を追加して kind: note（または history）を認識できるようにする
    - 例: `^\[[^\]]+\]\s*$`（FR引用）
    - 例: `^(Authority|Source):\s+`
  - structure で `section` の children に note を追加（paragraph/item/subitem には入れない）
  - NodeFactory で kind が note/history の場合は role="informative", normativity=None にする

## D. PIC/S profiles 同梱と最小テスト
- `src/qai_text2ir/profiles/` に以下を追加:
  - pics_part1_default_v1.yaml
  - pics_annex1_default_v1.yaml
- これらを `--parser-profile` で指定して parse_text_to_ir が動くことをテストする
- テストでは「ツリーができている」「代表的な num が入っている」ことだけを確認（細かい完全一致までは不要）
  - 例: chapter -> paragraph/item/point が生成される
  - 例: "5.1" や "5.1.1" や "i" が num に入る
